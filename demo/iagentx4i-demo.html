Security

Built-in, not bolted-on.

🛡️  "Worried about an AI modifying your production IBM i data? Here's why you don't need to be."
👤
Your Profile, Your Access

iAgentX uses your IBM i user profile via Code for IBM i's SSH session. No service account. No elevated privileges. No stored credentials. The AI operates within your own access rights.

🔍
Read-Only SQL, Always

Only SELECT, WITH, and VALUES are permitted. INSERT / UPDATE / DELETE / DDL are blocked at the extension layer — the AI literally cannot modify DB2 data.

No writes to DB2
📋
CL Command Allowlist

Only display/diagnostic CL verb prefixes are permitted. All other commands are blocked, even if the AI requests them.

DSP · LST · WRK · CHK · PRT · DMP · RTV · QRY
📝
No Blind Remote Writes

Editor write tools only operate on files already open in your VS Code. The AI cannot open and overwrite arbitrary source members without your involvement.

🌐
Loopback-Only HTTP Server

The MCP server binds exclusively to 127.0.0.1. It is completely unreachable from any other machine on the network — no firewall rules needed.

127.0.0.1 only
🔐
No Credential Storage

iAgentX stores absolutely nothing. SSH credentials live exclusively in Code for IBM i's secure credential store. Zero attack surface for credential theft.

Demo 1 — Job in Message Wait

Requirement: Diagnose a job stuck in MSGW

💬 "Is any of my jobs in message wait? Get the root cause."
Scenario
  • Batch job MSGWDEMO submitted via SBMJOB
  • Job silently stopped responding — no error, no output
  • Nobody knows why it's hanging or what it's waiting for
  • AI must find the job, check its status, and read its log
Prompt used
Is any of my jobs in message wait? Get the root cause.
🤖 Claude Code — Chat
Is any of my jobs in message wait? Get the root cause.
⚡ibmi_find_jobs — searching for MSGW jobs for current user
⚡ibmi_get_job_log — reading job log for MSGWDEMO
⚠ Job found in MSGW:
MSGWDEMO is waiting for an operator reply.

Root cause from job log:
The job completed two SQL steps successfully — scanned the system catalog and counted tables across schemas. At step 3, it sent a *INQ inquiry message to QSYSOPR:

"Nightly extract ready. Found 4,821 tables in 312 schemas. Reply G to proceed or C to cancel."

→ Job is healthy — it is waiting for a human decision before the extract runs. Reply in QSYSOPR to unblock it.
Tools used: ibmi_find_jobs ibmi_get_job_log

Demo 3 — Fix FK Hierarchy Function

Requirement: Debug & fix a SQL table function

💬 "Ever had a bug where your SQL function ignores all the FK relationships you carefully designed?"
Scenario
  • Function: XYZZ010788.schemaTableHierarchy('XYZLIBRARY')
  • Bug: Returns level = 1 for ALL tables
  • Expected: levels reflect FK depth (parent=1, child=2, grandchild=3…)
  • Goal: Fix it & deploy to library DK for testing
Prompt used
The function schemaTableHierarchy in my working directory is returning hierarchy level 1 for all tables when I run SELECT * FROM TABLE(XYZZ010788.schemaTableHierarchy('XYZLIBRARY')). The levels should reflect FK relationships. Investigate and fix it, then give me the fix to create in library DK and test against XYZLIBRARY.
✦ Key Insight
The AI reads the open source member directly from your editor — no copy-paste, no manual file transfer to the AI.
🤖 Claude Code — Chat
The function schemaTableHierarchy is returning level 1 for all tables. Investigate and fix it, then give me the fix for library DK.
⚡ibmi_get_active_editor — reading open QSQLSRC/SCHTBLHRC
⚡ibmi_run_sql — querying QSYS2.SYSKEYCST for FK relationships
🔍 Root Cause Found:
The recursive CTE uses CONSTRAINT_SCHEMA = p_schema but the FK catalog view uses UNIQUE_CONSTRAINT_SCHEMA for the referenced side. All FK joins are silently producing no rows — so every table falls back to level 1.

✅ Fix for library DK:
CREATE OR REPLACE FUNCTION DK.schemaTableHierarchy(p_schema VARCHAR(128))
RETURNS TABLE(TABLE_NAME VARCHAR(128), LEVEL INT)
-- [corrected JOIN condition: UNIQUE_CONSTRAINT_SCHEMA]
...
Ready to CREATE in library DK and test against XYZLIBRARY.
